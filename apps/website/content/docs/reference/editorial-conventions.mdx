---
title: Editorial conventions
description: Rules for keeping the public docs coherent, searchable, and version-ready as the site grows.
---

This website is the public documentation source of truth for better-age.

Package READMEs should stay concise and secondary. They can summarize a package and point back here, but they should not reimplement the full public narrative.

## What lives here

The website owns:

- product stance
- onboarding flow
- task-first guides
- integrations
- curated reference

## Information architecture rules

Keep the top-level docs model stable:

- Getting Started
- Guides
- Integrations
- Reference

When adding pages, prefer placing them into the existing model before inventing new top-level sections.

## Authoring rules

- keep docs hand-authored and product-honest
- prefer task-first guidance over generic feature tours
- use one page per CLI command or plugin surface in reference
- keep deep links precise and pages scannable
- avoid giant catch-all pages that will be painful to version later

## Source-of-truth rules

- this site owns the full public story
- package READMEs stay short
- internal planning docs can inform public docs, but are not the public docs themselves
- if wording conflicts, fix the site rather than letting duplicate narratives drift

## Search and AI/export rules

- write titles and descriptions so search results are useful
- preserve stable routes where possible
- keep `llms.txt` and page-level markdown export surfaces working
- avoid burying critical workflow terms behind vague marketing copy

## Reference publishing rules

For CLI and varlock reference:

- one page per command or surface when the behavior is distinct
- keep examples short and concrete
- document exact flags or contract details where they matter
- link back to guides when a user needs workflow context, not only syntax

## Versioning readiness rules

Live versioning is out of scope now, but content should stay easy to version later.

That means:

- prefer granular pages
- keep slugs stable
- avoid mixing unrelated topics into one page
- keep taxonomy changes intentional and rare
