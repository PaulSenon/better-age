---
title: Vision
description: The product bet behind better-age and the hard constraints that shape its UX.
---

`better-age` exists because the primitive is good, but the day to day UX still is not.

`age` is excellent crypto plumbing: small, legible, composable. But by itself it still leaves humans with too much ceremony around keys, sharing, rotation, and the practical handling of encrypted env files.

better-age takes the opposite move: narrower scope, fewer decisions, stronger defaults.

## Product bet

Encrypted env files should feel dead simple if the tool is opinionated enough.

That means:

- one obvious encrypted payload file
- one local identity setup
- one shareable identity string
- one explicit grant and revoke model
- one human path for reading secrets
- one machine path for loading secrets

## The intended team flow

1. Run `npx @better-age/cli setup` once.
2. Share your identity string from `bage me`.
3. Someone grants your identity on a payload.
4. From then on you can decrypt any payload shared with that identity.

## Hard constraints

Some choices are opinionated on purpose:

- key rotation is part of the model
- passphrase protection is mandatory
- key mode is `pq-hybrid`
- payload access is explicit
- the encrypted file stays visible and caller-owned

This project is willing to be less flexible to make the default path safer and easier to reason about.

## Non-goals

better-age is not trying to be:

- a generic file encryption tool
- a cloud secret manager
- a hidden secret registry
- a kitchen-sink wrapper around every `age` capability

It is intentionally small and biased toward one workflow: editing and sharing encrypted env files with a team.

## Why varlock matters

The varlock integration is not an afterthought. It is one of the reasons the project exists: keep local, explicit, composable env loading, but remove the no-setup-feeling pain around sharing encrypted env files across a team.
